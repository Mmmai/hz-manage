## CMDB 建模最佳实践

### 模型设计原则

#### 1. 单一职责原则

每个模型只描述一类资产，避免创建"万能模型"。

**好的设计**：
```
- 服务器模型：描述服务器属性
- 数据库模型：描述数据库属性
- 应用模型：描述应用属性
```

**不好的设计**：
```
- 资产模型：包含所有类型的资产（服务器、数据库、应用）
```

#### 2. 适度细化原则

模型字段数量建议控制在 20-30 个以内。

**字段过多时**：
- 考虑拆分模型
- 或使用字段分组
- 或使用 JSON 字段存储扩展信息

#### 3. 扩展性原则

为未来需求预留扩展空间：

- 预留扩展字段
- 使用 JSON 字段存储灵活数据
- 使用引用字段建立关联

### 模型命名规范

#### 模型名称

| 规范 | 示例 | 说明 |
|------|------|------|
| 小写英文 | `server` | 使用小写英文字母 |
| 下划线分隔 | `mysql_database` | 多词用下划线连接 |
| 单数形式 | `server`（非 `servers`） | 使用单数形式 |
| 简洁明了 | `database`（非 `db_instance`） | 名称应具有自描述性 |

#### 字段名称

| 规范 | 示例 | 说明 |
|------|------|------|
| 小写英文 | `hostname` | 使用小写英文字母 |
| 下划线分隔 | `ip_address` | 多词用下划线连接 |
| 具有描述性 | `cpu_cores`（非 `cpu`） | 名称应能描述字段含义 |
| 避免缩写 | `memory`（非 `mem`） | 避免使用不明确的缩写 |

#### 显示名称

| 规范 | 示例 | 说明 |
|------|------|------|
| 简洁中文 | `主机名` | 使用简洁的中文名称 |
| 包含单位 | `内存 (GB)` | 数值字段包含单位 |
| 保持一致 | 统一使用术语 | 保持命名风格一致 |

### 字段类型选择

| 业务场景 | 推荐类型 | 原因 |
|----------|----------|------|
| IP 地址 | `string` | 便于应用验证规则 |
| 主机名 | `string` | 单行文本，适合短文本 |
| 数量 | `integer` | 便于统计和计算 |
| 密码 | `password` | 自动加密，安全存储 |
| 关联 | `model_ref` | 建立模型关联，便于查询 |
| 描述 | `text` | 多行文本，适合长内容 |
| 标签 | `json` | 结构化数据，灵活扩展 |

## 数据质量管理

### 数据质量标准

#### 1. 完整性

- **必填字段**：关键信息设为必填
- **数据补全**：定期检查并补全缺失数据
- **数据验证**：创建时验证数据完整性

#### 2. 准确性

- **验证规则**：为关键字段配置验证规则
- **数据审核**：定期审核数据准确性
- **错误修正**：及时发现并修正错误数据

#### 3. 一致性

- **命名规范**：统一命名格式
- **数据格式**：统一日期、数字等格式
- **枚举值**：使用统一的枚举选项

#### 4. 唯一性

- **唯一约束**：为唯一字段设置约束
- **重复检查**：定期检查重复数据
- **数据清理**：及时清理重复数据

### 数据质量检查

#### 定期检查

1. **每周检查**
   - 必填字段缺失率
   - 数据格式错误率
   - 重复数据数量

2. **每月分析**
   - 数据质量趋势
   - 常见错误类型
   - 改进措施效果

3. **每季度评估**
   - 数据质量整体评估
   - 优化数据管理流程
   - 更新数据标准

#### 数据清理

1. **清理重复数据**
   - 识别重复实例
   - 合并或删除重复数据
   - 更新关联关系

2. **清理过期数据**
   - 识别过期实例
   - 移入空闲池
   - 归档或删除

3. **清理错误数据**
   - 识别格式错误数据
   - 修正或删除
   - 加强验证规则

## 权限管理建议

### 权限设计原则

#### 1. 最小权限原则

用户只拥有完成工作所需的最小权限。

**示例**：
- 只读用户：只有查看权限
- 运维人员：编辑实例权限
- 管理员：所有权限

#### 2. 职责分离原则

将关键操作分配给不同人员。

**示例**：
- 数据创建人员：只负责创建
- 数据审核人员：负责审核和批准
- 系统管理员：负责系统配置

#### 3. 按需授权原则

根据业务需要动态调整权限。

**示例**：
- 项目期间：临时授予项目权限
- 项目结束：及时收回权限

### 权限配置建议

#### 1. 角色权限

| 角色 | 权限说明 | 适用人员 |
|------|----------|----------|
| 只读用户 | 查看实例和字段 | 业务人员、管理层 |
- 运维人员 | 编辑实例、导入导出 | 运维工程师、开发人员 |
- 审计人员 | 查看审计日志 | 审计人员、安全管理员 |
- 管理员 | 所有权限 | 系统管理员、CMDB 负责人 |

#### 2. 数据权限

| 权限类型 | 说明 | 配置建议 |
|----------|------|----------|
| 模型权限 | 控制可访问的模型 | 按业务线分配 |
| 分组权限 | 控制可访问的分组 | 按环境或团队分配 |
- 字段权限 | 控制可查看的字段 | 敏感字段单独授权 |
- 操作权限 | 控制可执行的操作 | 按角色分配

#### 3. 审计权限

- 所有权限变更需要审批
- 记录权限授予和收回
- 定期审查用户权限

## 性能优化建议

### 查询优化

#### 1. 使用索引字段

优先使用有索引的字段查询：
- 实例名称
- 创建时间
- 唯一约束字段

#### 2. 缩小查询范围

- 先使用模型筛选
- 再使用字段条件
- 最后使用复杂查询

#### 3. 使用分页

- 大量结果使用分页
- 避免一次加载过多数据
- 提高页面响应速度

### 数据库优化

#### 1. 索引优化

- 为常用查询字段添加索引
- 为唯一约束字段添加索引
- 定期分析和优化索引

#### 2. 查询优化

- 避免全表扫描
- 使用分页查询
- 优化复杂查询

#### 3. 连接池优化

- 调整连接池大小
- 避免连接泄漏
- 监控连接使用情况

### 缓存优化

#### 1. 提高缓存命中率

- 延长缓存过期时间
- 预加载常用数据
- 优化缓存键设计

#### 2. 减少缓存大小

- 只缓存必要数据
- 压缩缓存数据
- 定期清理过期缓存

#### 3. 优化缓存刷新

- 使用增量刷新
- 异步刷新缓存
- 减少缓存失效范围

## 常见问题汇总（FAQ）

### 模型管理

**Q: 模型删除后还能恢复吗？**

A: 不能。模型删除会同时删除所有相关数据（字段、实例、关系、约束）。删除前请确认，或考虑先导出数据备份。

**Q: 如何修改已有字段的类型？**

A: 字段类型创建后不建议修改。如果必须修改：
1. 确保现有数据兼容新类型
2. 可能需要数据迁移
3. 建议创建新字段，迁移数据后删除旧字段

**Q: 实例名称模板可以后续修改吗？**

A: 可以。修改模板后：
- 已创建的实例名称不会自动更新
- 新创建的实例使用新模板
- 可以通过"更新实例名称"功能批量更新

### 实例管理

**Q: 为什么无法删除实例？**

A: 实例删除有以下限制：
1. 实例不在"空闲池"分组中
2. 实例存在关联关系

解决方法：
1. 先将实例移入"空闲池"
2. 删除所有关联关系

**Q: 导入失败怎么办？**

A: 导入失败通常由以下原因导致：
1. 数据格式不符合字段类型
2. 必填字段未填写
3. 引用字段的目标实例不存在
4. 违反唯一约束

查看错误信息，修正数据后重新导入。

**Q: 一个实例可以属于多个分组吗？**

A: 可以。实例与分组是多对多关系：
- 一个实例可以同时属于多个分组
- 例如：一台服务器可以同时在"生产环境"和"Web 集群"分组中

### 字段和验证

**Q: string 和 text 类型有什么区别？**

A:
- **string**：单行文本，适用于短文本（如主机名、IP）
- **text**：多行文本，适用于长文本（如描述、备注）
- string 类型的搜索性能更好

**Q: 密码字段如何查看明文？**

A:
1. 需要"显示密码"权限
2. 点击"显示密码"按钮
3. 系统解密后显示明文
4. 操作会被审计日志记录

**Q: 验证规则可以修改吗？**

A: 可以修改，但需要注意：
- 内置规则（built_in=true）不可删除
- 修改规则后会影响所有使用该规则的字段
- 建议先创建新规则，再迁移字段引用

### 关系和审计

**Q: 关系和字段引用有什么区别？**

A:
- **关系**：独立的关系实例，支持拓扑可视化，可以携带属性
- **字段引用**：模型字段（model_ref 类型），存储在实例中

建议：
- 需要可视化：使用关系
- 简单引用：使用字段引用

**Q: 审计记录可以修改吗？**

A: 不可以。审计记录设计为不可修改：
- 防止篡改历史
- 确保审计可信
- 满足合规要求

### 系统维护

**Q: 什么时候需要清理缓存？**

A: 以下情况需要清理缓存：
- 数据不一致时
- 系统更新后
- 性能问题时
- 数据迁移后

大多数情况下系统会自动刷新缓存，不需要手动清理。

**Q: 系统卡顿怎么办？**

A: 排查步骤：
1. 查看系统资源使用情况
2. 查看应用日志和错误日志
3. 尝试清理缓存
4. 检查数据库负载
5. 联系系统管理员

## 故障排查指南

### 数据不一致

**症状**：界面显示的数据与实际不符

**排查步骤**：
1. 清理缓存
2. 检查数据库数据
3. 查看审计日志
4. 检查并发更新

**解决方法**：
- 清理缓存刷新数据
- 根据审计日志恢复数据
- 优化并发更新逻辑

### 性能问题

**症状**：系统响应慢，查询超时

**排查步骤**：
1. 检查系统资源使用情况
2. 检查缓存命中率
3. 检查数据库性能
4. 查看慢查询日志

**解决方法**：
- 清理或优化缓存
- 添加数据库索引
- 优化查询语句
- 扩容系统资源

### 权限问题

**症状**：用户无法访问某些功能或数据

**排查步骤**：
1. 检查用户角色和权限
2. 检查数据权限配置
3. 查看权限审计日志

**解决方法**：
- 调整用户权限
- 修改数据权限配置
- 检查权限继承关系

### 导入导出问题

**症状**：导入失败或导出数据不完整

**排查步骤**：
1. 检查数据格式
2. 查看错误信息
3. 检查字段配置
4. 检查引用数据

**解决方法**：
- 修正数据格式
- 补全必填字段
- 创建引用实例
- 分批导入数据
