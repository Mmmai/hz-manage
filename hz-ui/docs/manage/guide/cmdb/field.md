title: 字段配置
---

## 概述

字段（Field）是模型的基本组成单元，定义了资产实例的属性结构。CMDB 支持 11 种字段类型，每种类型都有其特定的用途和验证规则。

## 字段类型详解

### 1. string（字符串）

单行文本字段，适用于短文本内容。

**适用场景**：
- 主机名、IP 地址、MAC 地址
- 设备序列号、资产编号
- 版本号、标识符

**示例配置**：
```
字段名：hostname
显示名：主机名
类型：string
是否必填：是
验证规则：正则 ^[a-z0-9-]+$
默认值：-
```

### 2. text（文本）

多行文本字段，适用于长文本内容。

**适用场景**：
- 描述、备注、说明
- 配置内容
- 日志记录

**示例配置**：
```
字段名：description
显示名：描述
类型：text
是否必填：否
```

### 3. boolean（布尔值）

真/假字段，适用于二元选择。

**适用场景**：
- 是否启用、是否高可用
- 是否生产环境、是否监控
- 开关状态

**示例配置**：
```
字段名：enabled
显示名：是否启用
类型：boolean
是否必填：是
默认值：true
```

::: tip 布尔值表示
在界面中以开关或复选框形式展示，值为 `true` 或 `false`。
:::

### 4. enum（枚举）

下拉选择字段，从预定义的选项中选择。

**适用场景**：
- 操作系统类型、环境分类
- 状态、类型、级别
- 有限的固定选项

**示例配置**：
```
字段名：os_type
显示名：操作系统
类型：enum
是否必填：是
验证规则：枚举
```

**枚举值配置**（在验证规则中）：
```json
{
  " CentOS 7": "CentOS 7",
  "CentOS 8": "CentOS 8",
  "Ubuntu 20.04": "Ubuntu 20.04",
  "Ubuntu 22.04": "Ubuntu 22.04"
}
```

### 5. json（JSON 数据）

结构化数据字段，存储 JSON 格式的数据。

**适用场景**：
- 标签（Tags）
- 扩展属性
- 配置信息

**示例配置**：
```
字段名：tags
显示名：标签
类型：json
是否必填：否
```

**数据示例**：
```json
{
  "env": "production",
  "team": "backend",
  "cost_center": "IT001"
}
```

::: tip JSON 字段验证
JSON 字段会自动验证格式，确保输入的是合法的 JSON。
:::

### 6. integer（整数）

整数字段，适用于数量、编号等。

**适用场景**：
- CPU 核数、内存大小
- 端口号、数量
- 优先级、权重

**示例配置**：
```
字段名：cpu_cores
显示名：CPU 核数
类型：integer
是否必填：是
单位：核
验证规则：范围 1-128
```

### 7. float（浮点数）

小数字段，适用于需要精确小数的场景。

**适用场景**：
- 磁盘容量、带宽
- 百分比、比率
- 价格、成本

**示例配置**：
```
字段名：disk_size
显示名：磁盘大小
类型：float
是否必填：是
单位：GB
验证规则：范围 0-10240
```

### 8. password（密码）

加密密码字段，使用国密算法加密存储。

**适用场景**：
- 数据库密码
- SSH 密码
- API 密钥

**示例配置**：
```
字段名：db_password
显示名：数据库密码
类型：password
是否必填：否
```

::: warning 密码安全
- 密码字段使用国密算法加密后存储
- 界面显示时自动脱敏
- 导出时不包含密码值
- 查看/显示密码需要特定权限
:::

### 9. date（日期）

日期字段，存储日期信息。

**适用场景**：
- 购买日期、到期日期
- 维护日期
- 上线日期

**示例配置**：
```
字段名：purchase_date
显示名：购买日期
类型：date
是否必填：否
```

**支持的日期格式**：
- `YYYY-MM-DD`（推荐）：2024-03-21
- `YYYY/MM/DD`：2024/03/21
- `DD-MM-YYYY`：21-03-2024

### 10. datetime（日期时间）

日期时间字段，存储精确到秒的时间信息。

**适用场景**：
- 创建时间、更新时间
- 最后维护时间
- 过期时间

**示例配置**：
```
字段名：last_maintenance
显示名：最后维护时间
类型：datetime
是否必填：否
```

**支持的日期时间格式**：
- `YYYY-MM-DD HH:MM:SS`（推荐）：2024-03-21 15:30:00
- `YYYY/MM/DD HH:MM:SS`：2024/03/21 15:30:00
- ISO 8601：2024-03-21T15:30:00

### 11. model_ref（模型引用）

引用字段，关联到其他模型的实例。

**适用场景**：
- 服务器所属机房
- 应用部署的服务器
- 数据库依赖的中间件

**示例配置**：
```
字段名：datacenter
显示名：所属机房
类型：model_ref
引用模型：机房
是否必填：是
```

::: tip 引用字段的作用
引用字段建立了模型间的关联关系，可以：
- 在拓扑图中可视化展示
- 在关系查询中追踪依赖
- 实现级联查询和统计
:::

## 字段属性说明

| 属性 | 类型 | 说明 | 示例 |
|------|------|------|------|
| name | string | 字段唯一标识，英文 | `hostname` |
| verbose_name | string | 字段显示名称，中文 | `主机名` |
| type | string | 字段数据类型 | `string` |
| required | boolean | 是否必填 | `true` |
| editable | boolean | 是否可编辑 | `true` |
| default | string | 默认值 | `Linux` |
| validation_rule | foreign key | 验证规则 | IPv4 验证 |
| unit | string | 数值单位 | `GB` |
| description | text | 字段描述 | 用于标识主机的名称 |
| order | integer | 显示顺序 | `1` |

## 字段类型与验证规则映射

不同字段类型支持的验证规则：

| 字段类型 | 支持的验证规则 |
|----------|----------------|
| string | 正则、长度、IP、IPv4、IPv6、URL、Email、Phone |
| text | 正则、长度 |
| boolean | 布尔值 |
| enum | 枚举值 |
| json | JSON 格式 |
| integer | 范围 |
| float | 范围 |
| password | 密码规则 |
| date | 日期格式 |
| datetime | 日期时间格式 |
| model_ref | 模型关联 |

详细配置请参考 [校验配置](./validation.md)。

## 最佳实践

### 命名规范

**字段名称**：
- 使用英文小写字母和下划线
- 名称应具有自描述性
- 避免使用数据库关键字

| 好的命名 | 不好的命名 | 说明 |
|----------|------------|------|
| `hostname` | `name` | 太通用 |
| `ip_address` | `ip` | 应完整 |
| `cpu_cores` | `cpu` | 不明确 |
| `datacenter_id` | `dc` | 使用缩写 |

**显示名称**：
- 使用简洁的中文
- 包含必要的单位
- 保持一致性

| 好的命名 | 不好的命名 |
|----------|------------|
| `主机名` | `名` |
| `IP 地址` | `IP` |
| `CPU 核数` | `核数` |
| `内存大小 (GB)` | `内存` |

### 类型选择建议

1. **IP 地址使用 string 类型**
   - 便于应用验证规则
   - 支持 IPv4 和 IPv6
   - 便于格式化显示

2. **数量使用 integer 类型**
   - CPU 核数、内存大小
   - 便于统计和计算
   - 支持范围验证

3. **敏感信息使用 password 类型**
   - 数据库密码、SSH 密码
   - 自动加密存储
   - 显示时脱敏

4. **关联使用 model_ref 类型**
   - 服务器关联机房
   - 应用关联服务器
   - 便于查询和拓扑展示

### 字段设计原则

1. **最小化原则**
   - 只添加必要的字段
   - 避免冗余字段
   - 可选字段设为非必填

2. **原子性原则**
   - 字段值不可再分
   - 避免组合字段（如 `location` 拆分为 `building`, `floor`, `room`）

3. **一致性原则**
   - 相同含义使用相同字段名
   - 相同类型使用相同验证规则
   - 统一单位和格式

4. **扩展性原则**
   - 预留扩展字段
   - 或使用 JSON 字段存储灵活数据
   - 考虑未来需求变化

## 常见问题

### Q: string 和 text 类型有什么区别？

A:
- **string**：单行文本，适用于短文本（如主机名、IP）
- **text**：多行文本，适用于长文本（如描述、备注）
- string 类型的搜索性能更好

### Q: 如何存储带单位的数值？

A:
- 方式一：数值字段 + 单位属性（推荐）
  ```
  字段名：memory
  类型：integer
  单位：GB
  值：16
  ```

- 方式二：字符串字段
  ```
  字段名：memory
  类型：string
  值：16GB
  ```
  不推荐：无法进行数值比较和计算

### Q: model_ref 字段可以引用多个模型吗？

A: 不可以。一个 model_ref 字段只能引用一个模型。如果需要引用多种类型：
- 创建多个字段
- 或使用枚举字段 + 多个引用字段
- 或考虑重新设计模型结构

### Q: 密码字段如何查看明文？

A:
- 需要"显示密码"权限
- 点击"显示密码"按钮
- 系统解密后显示明文
- 操作会被审计日志记录

### Q: JSON 字段如何查询？

A:
- JSON 字段支持精确匹配查询
- 暂不支持 JSON 内部字段的查询
- 如果需要查询，考虑将重要字段提取为独立字段

### Q: 字段创建后可以修改类型吗？

A: 可以，但需要注意：
- 确保现有数据兼容新类型
- 可能需要数据迁移
- 建议先导出备份
- 不兼容的数据可能导致丢失
