title: 唯一性约束
---

## 概述

唯一性约束（Unique Constraint）用于确保模型实例中一个或多个字段的组合值在模型内唯一，防止数据重复。

### 约束的价值

- **数据唯一性**：确保关键字段不重复
- **业务规则**：实现"一对一"或"不重复"的业务约束
- **数据质量**：防止重复数据进入系统
- **查询优化**：唯一字段可以提升查询性能

## 内置约束

系统会自动为每个模型创建以下内置约束：

### 实例名称唯一约束

- **约束字段**：`model` + `instance_name`
- **说明**：同一模型内的实例名称必须唯一
- **内置标识**：built_in=true
- **可删除**：否

::: tip 实例名称唯一性
实例名称在模型内唯一，但不同模型的实例可以有相同的名称。
:::

## 创建唯一约束

### 单字段唯一约束

确保单个字段的值在模型内唯一。

**操作步骤**：

1. 进入 **模型管理** 页面
2. 选择目标模型
3. 切换到 **唯一约束** 标签
4. 点击 **新增约束**
5. 填写约束信息：
   - **约束字段**：选择要约束的字段
   - **验证空值**：是否验证空值（默认 false）
   - **描述**：约束说明（可选）
6. 点击 **保存**

**示例场景**：

| 字段 | 约束说明 |
|------|----------|
| IP 地址 | 服务器 IP 必须唯一 |
| 序列号 | 设备序列号必须唯一 |
| 域名 | 域名必须唯一 |

### 组合唯一约束

确保多个字段的组合值在模型内唯一。

**操作步骤**：

1. 进入 **模型管理** 页面
2. 选择目标模型
3. 切换到 **唯一约束** 标签
4. 点击 **新增约束**
5. 填写约束信息：
   - **约束字段**：选择多个字段（按住 Ctrl 多选）
   - **验证空值**：是否验证空值（默认 false）
   - **描述**：约束说明（可选）
6. 点击 **保存**

**示例场景**：

| 字段组合 | 约束说明 |
|----------|----------|
| 机房 + 机柜 + 机位 | 同一机房内的机柜机位不能重复 |
| 应用 + 环境 | 同一环境下的应用名不能重复 |
| 数据库 + 端口 | 同一服务器上的数据库端口不能重复 |

## 约束配置说明

### 约束字段（fields）

参与唯一约束的字段列表。

- **单字段**：`["ip_address"]`
- **多字段**：`["datacenter", "cabinet", "position"]`

### 验证空值（validate_null）

是否对空值进行唯一性验证。

- **false**（默认）：空值不参与唯一性验证，允许多个空值
- **true**：空值也参与唯一性验证，最多一个空值

::: tip 空值处理建议
- 对于非必填字段，通常设置 validate_null=false
- 对于必填字段，空值会被必填验证拦截，唯一约束无需处理
:::

## 业务场景示例

### 场景一：服务器 IP 唯一

**需求**：确保服务器的 IP 地址不重复。

**配置**：
```
约束字段：ip_address
验证空值：false
描述：服务器 IP 地址必须唯一
```

**结果**：
- 同一模型内，IP 地址不能重复
- 两个实例不能有相同的 IP
- 如果 IP 为空，允许多个空值实例

### 场景二：设备序列号唯一

**需求**：确保设备的序列号（SN）不重复。

**配置**：
```
约束字段：serial_number
验证空值：false
描述：设备序列号必须唯一
```

### 场景三：机房内机柜位置唯一

**需求**：同一机房内的机柜和机位组合不能重复。

**配置**：
```
约束字段：[datacenter, cabinet, position]
验证空值：false
描述：同一机房内的机柜机位不能重复
```

**结果**：
- 北京机房-01柜-01U：可以
- 北京机房-01柜-02U：可以
- 北京机房-01柜-01U：重复（不允许）
- 上海机房-01柜-01U：可以（不同机房）

### 场景四：应用在不同环境下可重名

**需求**：应用名在不同环境下可以重复，但同一环境下不能重复。

**配置**：
```
约束字段：[app_name, environment]
验证空值：false
描述：同一环境下的应用名不能重复
```

**结果**：
- 电商网站-生产环境：可以
- 电商网站-测试环境：可以
- 电商网站-生产环境：重复（不允许）

## 约束删除

### 删除约束

1. 进入 **模型管理** 页面
2. 选择目标模型
3. 切换到 **唯一约束** 标签
4. 找到要删除的约束
5. 点击 **删除** 按钮
6. 确认删除操作

::: danger 内置约束限制
内置约束（built_in=true）不可删除。实例名称唯一约束是内置约束。
:::

::: warning 删除约束的影响
删除约束后：
- 现有数据不受影响
- 新创建的实例不再受该约束限制
- 可能导致数据重复
:::

## 最佳实践

### 约束设计原则

1. **必要性原则**
   - 只为真正需要唯一的字段创建约束
   - 避免过多约束影响性能
   - 考虑业务实际需求

2. **合理组合**
   - 组合约束的字段应有业务关联
   - 避免过度组合导致约束过严
   - 示例：
     - 好的组合：`[机房, 机柜, 机位]`（物理位置）
     - 不好的组合：`[IP, 主机名, 系统]`（无业务关联）

3. **空值处理**
   - 非必填字段通常设置 validate_null=false
   - 必填字段的空值会被必填验证拦截
   - 考虑业务是否允许空值

### 常见业务场景

| 业务场景 | 约束字段组合 |
|----------|--------------|
| 服务器 IP 唯一 | `[ip_address]` |
| 设备 SN 唯一 | `[serial_number]` |
| 域名唯一 | `[domain_name]` |
| 机房内位置唯一 | `[datacenter, cabinet, position]` |
| 环境内应用名唯一 | `[app_name, environment]` |
| 服务器内数据库端口唯一 | `[server_id, port]` |

### 性能考虑

1. **约束数量**
   - 每个模型建议不超过 5 个唯一约束
   - 过多约束会影响插入性能

2. **字段选择**
   - 优先选择索引字段（如主键、唯一字段）
   - 避免对长文本字段创建约束

3. **组合约束**
   - 组合字段数量建议不超过 3 个
   - 字段越多，约束检查越慢

## 常见问题

### Q: 唯一约束和唯一索引有什么区别？

A:
- **唯一约束**：业务层面的数据完整性保证
- **唯一索引**：数据库层面的查询优化

CMDB 的唯一约束在应用层实现，不依赖数据库索引。

### Q: 为什么内置约束不能删除？

A: 内置约束是系统的基础规则：
- 实例名称唯一约束是系统核心逻辑
- 删除会导致系统功能异常
- 这是系统稳定性保护机制

### Q: validate_null 参数的作用是什么？

A: 控制空值是否参与唯一性验证：
- `false`：空值不参与验证，允许多个空值（推荐）
- `true`：空值也参与验证，最多一个空值

对于非必填字段，通常设置为 false。

### Q: 唯一约束可以跨模型生效吗？

A: 不可以。唯一约束只在模型内生效：
- 服务器模型的 IP 约束不影响数据库模型的 IP
- 如需跨模型唯一，需要业务层处理
- 或在应用层面通过唯一标识符实现

### Q: 如何处理历史数据中的重复数据？

A: 创建唯一约束前：
1. 先查询并清理重复数据
2. 删除或合并重复的实例
3. 再创建唯一约束

如果已有重复数据，创建约束时会失败。

### Q: 组合唯一约束的字段顺序重要吗？

A: 从功能角度不重要，但从性能角度有影响：
- 系统会按顺序检查字段
- 将区分度高的字段放在前面
- 示例：`[环境, 应用名]` 优于 `[应用名, 环境]`（如果环境种类较少）

### Q: 唯一约束如何影响导入操作？

A:
- 导入时会进行唯一性验证
- 如果违反约束，整批导入都会失败
- 错误信息会指出哪些记录违反了约束
- 建议导入前先清理重复数据
