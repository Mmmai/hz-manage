## 功能概述

关联关系定义模型和实例之间的引用关系，支持建立资产间的拓扑连接，实现可视化的关系管理。

### 关系的价值

- **拓扑可视化**：通过关系图展示资产间的连接
- **依赖分析**：分析资产的上下游依赖
- **影响评估**：评估变更对关联资产的影响
- **故障排查**：快速定位故障关联路径

## 关系定义管理

### 关系定义概念

关系定义（Relation Definition）描述两个模型间的关系类型和规则：

- **关系名称**：关系的唯一标识
- **拓扑类型**：关系的图结构类型
- **源模型**：关系起始端的模型
- **目标模型**：关系终止端的模型
- **关系动词**：描述关系的动作或状态

### 拓扑类型

| 类型 | 说明 | 示例 |
|------|------|------|
| directed | 有向图 | 服务器 → 机房（单向关联） |
| undirected | 无向图 | 服务器 ↔ 交换机（双向连接） |
| daggered | 有向无环图 | 应用 → 数据库（无环依赖） |

### 创建关系定义

1. 进入 **关联关系** 页面
2. 切换到 **关系定义** 标签
3. 点击 **新增关系定义**
4. 填写关系定义信息：
   - **关系名称**：关系的唯一标识，如 `deployed_on`
   - **拓扑类型**：选择图结构类型
   - **正向动词**：从源到目标的关系描述，如 `部署在`
   - **反向动词**：从目标到源的关系描述，如 `承载`
   - **源模型**：选择允许作为源端的模型
   - **目标模型**：选择允许作为目标端的模型
   - **属性模式**（可选）：定义关系的附加属性
   - **描述**：关系说明（可选）
5. 点击 **保存**

::: tip 关系命名规范
- 使用英文小写字母和下划线
- 采用 `动词` 或 `动词_名词` 格式
- 示例：`deployed_on`, `depends_on`, `connected_to`
:::

### 内置关系

系统预置了以下关系定义：

| 关系名称 | 正向动词 | 反向动词 | 拓扑类型 |
|----------|----------|----------|----------|
| deployed_on | 部署在 | 承载 | daggered |
| depends_on | 依赖于 | 被依赖 | daggered |
| connected_to | 连接到 | 被连接 | undirected |

::: warning 内置关系限制
内置关系（built_in=true）不可删除，但可以编辑和添加允许的模型。
:::

### 关系属性模式

关系可以携带额外的属性信息：

```json
{
  "port": "端口号",
  "protocol": "协议",
  "bandwidth": "带宽",
  "create_time": "创建时间"
}
```

**示例**：
- 应用连接到数据库的关系，可以包含端口号、协议等属性

## 关系实例管理

### 关系实例概念

关系实例（Relation Instance）是具体实例间的关联：

- **源实例**：关系起始端的实例
- **目标实例**：关系终止端的实例
- **关系定义**：引用的关系定义
- **关系属性**：关系的附加属性值

### 创建关系实例

1. 进入 **关联关系** 页面
2. 切换到 **关系实例** 标签
3. 点击 **新增关系**
4. 填写关系信息：
   - **关系类型**：选择关系定义
   - **源实例**：选择源端实例
   - **目标实例**：选择目标端实例
   - **关系属性**：填写属性值（可选）
5. 点击 **保存**

::: tip 关系唯一性
同一个源实例和目标实例之间，同一种关系类型只能有一个关系实例。
:::

### 批量创建关系

1. 进入 **关联关系** 页面
2. 切换到 **关系实例** 标签
3. 点击 **批量创建**
4. 填写批量创建信息：
   - **关系类型**：选择关系定义
   - **源实例**：选择多个源实例
   - **目标实例**：选择目标实例
   - **关系属性**：填写属性值（可选）
5. 点击 **保存**

**适用场景**：
- 多个服务器部署在同一机房
- 多个应用依赖同一个数据库
- 多个设备连接到同一个交换机

### 删除关系实例

1. 在关系实例列表中找到目标关系
2. 点击 **删除** 按钮
3. 确认删除操作

::: warning 删除关系的影响
删除关系不会删除实例：
- 源实例和目标实例仍然存在
- 只是删除了它们之间的关联
- 拓扑图中不再显示该关系
:::

## 拓扑可视化

### 查看拓扑图

1. 进入 **关联关系** 页面
2. 选择目标实例
3. 点击 **查看拓扑** 按钮
4. 系统生成拓扑图

### 拓扑图元素

- **节点**：资产实例
  - 节点大小：可根据重要性调整
  - 节点颜色：可根据状态或类型着色
  - 节点图标：使用模型图标

- **连线**：关联关系
  - 连线方向：箭头指示关系方向
  - 连线标签：显示关系动词
  - 连线粗细：可表示关系强度

### 布局算法

系统支持多种拓扑布局算法：

| 算法 | 说明 | 适用场景 |
|------|------|----------|
| 力导向 | 节点间有斥力，连接有拉力 | 一般关系图 |
| 层次布局 | 按层次排列节点 | 树形结构、依赖关系 |
| 圆形布局 | 节点排列成圆形 | 环形拓扑 |
| 网格布局 | 节点排列成网格 | 规则结构 |

### 交互操作

- **缩放**：鼠标滚轮缩放拓扑图
- **拖拽**：拖拽节点调整位置
- **点击节点**：查看节点详情和关联关系
- **点击连线**：查看关系详情
- **筛选**：按关系类型筛选显示

::: tip 拓扑图性能
拓扑图节点数量过多时可能影响性能：
- 建议显示的节点数量不超过 100 个
- 使用筛选条件缩小范围
- 或使用分层展示
:::

## 业务场景示例

### 场景一：应用部署关系

**需求**：展示应用部署在哪些服务器上。

**配置**：
```
关系定义：
- 关系名称：deployed_on
- 正向动词：部署在
- 反向动词：承载
- 拓扑类型：daggered（有向无环图）
- 源模型：应用
- 目标模型：服务器

关系实例：
- 源实例：电商网站（应用）
- 目标实例：web-server-01（服务器）
- 关系类型：deployed_on
```

**拓扑图**：
```
电商网站 --部署在--> web-server-01
                      |
                      +-- web-server-02
```

### 场景二：应用依赖关系

**需求**：展示应用依赖的数据库、中间件等。

**配置**：
```
关系定义：
- 关系名称：depends_on
- 正向动词：依赖于
- 反向动词：被依赖
- 拓扑类型：daggered（有向无环图）
- 源模型：应用
- 目标模型：数据库、中间件

关系实例：
- 源实例：电商网站（应用）
- 目标实例：MySQL 主库（数据库）
- 关系类型：depends_on
- 属性：{端口: 3306, 协议: TCP}
```

**拓扑图**：
```
电商网站 --依赖于--> MySQL 主库
         |
         +--> Redis 缓存
```

### 场景三：网络连接关系

**需求**：展示服务器与交换机的连接关系。

**配置**：
```
关系定义：
- 关系名称：connected_to
- 正向动词：连接到
- 反向动词：被连接
- 拓扑类型：undirected（无向图）
- 源模型：服务器
- 目标模型：网络设备

关系实例：
- 源实例：web-server-01（服务器）
- 目标实例：核心交换机-01（网络设备）
- 关系类型：connected_to
- 属性：{端口: eth0, VLAN: 100}
```

**拓扑图**：
```
web-server-01 <----连接----> 核心交换机-01
     ^
     |
     连接
     |
db-server-01
```

### 场景四：物理位置关系

**需求**：展示服务器的物理位置（机房、机柜、机位）。

**配置**：
```
关系定义：
- 关系名称：located_in
- 正向动词：位于
- 反向动词：包含
- 拓扑类型：directed（有向图）
- 源模型：服务器
- 目标模型：机房、机柜

关系实例：
- 源实例：web-server-01（服务器）
- 目标实例：北京机房-01柜-03U（机柜）
- 关系类型：located_in
```

**拓扑图**：
```
web-server-01 --位于--> 北京机房-01柜
                       |
                       +-- web-server-02 --位于--> 北京机房-02柜
```

## 最佳实践

### 关系设计原则

1. **明确关系语义**
   - 关系动词应清晰描述关系
   - 避免使用模糊的关系
   - 示例：`部署在` 优于 `关联`

2. **选择合适的拓扑类型**
   - 有向无环图（daggered）：依赖关系、部署关系
   - 无向图（undirected）：连接关系、对等关系
   - 有向图（directed）：包含关系、位置关系

3. **控制关系数量**
   - 避免创建过多的关系定义
   - 相似关系可以合并
   - 示例：不需要为每种数据库创建单独的关系

4. **使用关系属性**
   - 重要信息存储在关系属性中
   - 示例：端口号、协议、带宽
   - 避免在关系名称中包含可变信息

### 关系命名规范

**关系名称**：
- 使用英文小写字母和下划线
- 采用 `动词` 或 `动词_介词` 格式
- 示例：
  - `deployed_on`（部署在）
  - `depends_on`（依赖于）
  - `connected_to`（连接到）
  - `located_in`（位于）

**关系动词**：
- 正向动词：从源到目标的描述
- 反向动词：从目标到源的描述
- 示例：
  - 正向：`部署在`，反向：`承载`
  - 正向：`依赖于`，反向：`被依赖`

### 拓扑图优化

1. **节点筛选**
   - 只显示相关的节点
   - 隐藏不重要的节点
   - 使用分组或折叠功能

2. **颜色编码**
   - 按模型类型着色
   - 按实例状态着色
   - 按重要性着色

3. **布局选择**
   - 依赖关系：使用层次布局
   - 网络拓扑：使用力导向布局
   - 环形结构：使用圆形布局

## 常见问题

### Q: 关系定义可以删除吗？

A:
- 自定义关系可以删除
- 内置关系（built_in=true）不可删除
- 删除关系定义会同时删除所有关系实例

### Q: 同一实例间可以有多个关系吗？

A: 可以：
- 同一实例间可以有不同类型的关系
- 例如：应用 A 和服务器 B 可以同时有 `部署在` 和 `连接到` 关系
- 但同一种关系类型只能有一个

### Q: 关系可以跨模型吗？

A: 可以。关系定义的源模型和目标模型可以是：
- 同一模型（自关联）
- 不同模型（跨模型关联）
- 多个模型（多对多）

### Q: 拓扑图如何导出？

A:
- 系统支持导出拓扑图为图片
- 或导出拓扑数据为 JSON
- 具体导出方式取决于界面功能

### Q: 关系属性可以修改吗？

A: 可以：
- 编辑关系实例
- 修改关系属性值
- 保存更改

### Q: 如何查找实例的所有关联？

A:
1. 在实例详情页查看关联关系
2. 或在关联关系页面筛选该实例
3. 或在拓扑图中查看该实例的连接

### Q: 关系和字段引用有什么区别？

A:
- **关系**：独立的关系实例，支持拓扑可视化，可以携带属性
- **字段引用**：模型字段（model_ref 类型），存储在实例中

建议：
- 需要可视化：使用关系
- 简单引用：使用字段引用
